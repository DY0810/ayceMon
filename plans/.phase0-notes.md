### Next.js 16 facts that invalidate copy-pasted Supabase snippets

Confirmed from the docs above (with quoted citations so Phases 1/2/3 cannot accidentally regress):

1. **`middleware.ts` has been renamed to `proxy.ts`.**
   - `proxy.md` line 11: *"The `middleware` file convention is deprecated and has been renamed to `proxy`. See [Migration to Proxy] for more details."*
   - `proxy.md` line 770 (Version history): *"v16.0.0: Middleware is deprecated and renamed to Proxy"*
   - The file lives at the repo root alongside `app/` (not inside `app/`). Also valid inside `src/` if you have one.
   - Signature: `export function proxy(request: NextRequest)` (or `export default function proxy(...)`). May be marked `async`.
   - Optional `export const config = { matcher: [...] }`.
   - Codemod available: `npx @next/codemod@canary middleware-to-proxy .` (we are writing `proxy.ts` from scratch, not migrating).

2. **`cookies()` is async.**
   - `cookies.md` line 6: *"`cookies` is an **async** function that allows you to read the HTTP incoming request cookies…"*
   - `cookies.md` line 67: *"`cookies` is an **asynchronous** function that returns a promise. You must use `async/await` or React's `use` function to access cookies."*
   - Canonical call: `const cookieStore = await cookies()`.
   - Note: `cookies.md` line 68 warns the Next 15 sync fallback is deprecated — do NOT rely on it.

3. **Dynamic `params` in PageProps / LayoutProps / Route Handlers are async.**
   - `layout.md` line 70: `params: Promise<{ team: string }>`
   - `layout.md` line 77: `export default async function Layout({ children, params }) { const { team } = await params; ... }`
   - Same for `page.tsx`, `route.ts`, and `generateMetadata`. Every destructure **must** await.

4. **`headers()` and `draftMode()` follow the same async rule as `cookies()`** (per Next 15 async-dynamic-APIs codemod). Phases that use `headers()` must also `await` it.

### Google Places API (New) — endpoints, field masks, SKUs

Sources:
- https://developers.google.com/maps/documentation/places/web-service/place-autocomplete
- https://developers.google.com/maps/documentation/places/web-service/place-details
- https://developers.google.com/maps/documentation/places/web-service/usage-and-billing
- https://developers.google.com/maps/documentation/places/web-service/session-pricing
- https://developers.google.com/maps/billing-and-pricing/pricing (pricing table)

#### Place Autocomplete (New)

- **Method & URL:** `POST https://places.googleapis.com/v1/places:autocomplete`
- **Required headers:**
  - `Content-Type: application/json`
  - `X-Goog-Api-Key: <GOOGLE_PLACES_API_KEY>`
  - `X-Goog-FieldMask: suggestions.placePrediction.text.text,suggestions.placePrediction.placeId`
- **Example request body:**
  ```json
  {
    "input": "pizza near me",
    "sessionToken": "<client-generated UUID, unique per session>",
    "includeQueryPredictions": false
  }
  ```
- **Session token rules (verbatim from usage-and-billing):** *"A session token is good for one session and cannot be used for more than one session. If you reuse a session token, the session is considered invalid and the requests are charged as if no session token were provided."* Sessions *"must be unique per project"*.
- **Session token lifetime:** The docs do **not** state a specific duration (no "3 minutes" etc.). A session starts at the first Autocomplete+token request, continues for any subsequent Autocomplete requests with the same token, and terminates when a Place Details call uses the token. After termination, reuse is billed as if no token was provided.

#### Place Details (New)

- **Method & URL:** `GET https://places.googleapis.com/v1/places/{PLACE_ID}`
- **Required headers:**
  - `X-Goog-Api-Key: <GOOGLE_PLACES_API_KEY>`
  - `X-Goog-FieldMask: id,displayName,formattedAddress,location`
- **SKU tier for the chosen field mask:** The field mask `id,displayName,formattedAddress,location` bills at **Place Details Pro**, because `displayName` is a Pro-tier field. Per usage-and-billing docs: *"You are then billed at the highest SKU applicable to your request."* `id`/`formattedAddress`/`location` alone would be Essentials; adding `displayName` bumps the whole call to Pro.

#### Places pricing (Places API New)

Source: https://developers.google.com/maps/billing-and-pricing/pricing

| SKU | Free cap | $/1000 (cap–100k) | Next tier (100k–500k) |
|---|---|---|---|
| Place Details Pro | 5,000 | **$17.00** | $13.60 |
| Place Details Essentials | 10,000 | **$5.00** | $4.00 |
| Place Details Essentials IDs-Only | unlimited | — | — |
| Autocomplete Requests (per-request / abandoned) | 10,000 | **$2.83** | $2.27 |
| Autocomplete Session Usage (per-session, terminated by Details) | unlimited | **free** | free |

Session behavior (session-pricing page, verbatim):
- **Session ending in Place Details Essentials:** *"The first 12 Autocomplete (New) requests are billed at the SKU: Autocomplete Requests. Any remaining Autocomplete (New) requests are billed at the SKU: Autocomplete Session Usage, meaning there is no charge for those requests."*
- **Session ending in Place Details Pro or Enterprise (our case):** *"All Autocomplete (New) requests are billed at the SKU: Autocomplete Session Usage, meaning there is no charge for those requests."* (Only the terminating Place Details Pro call is billed.)
- **Abandoned session (no Place Details):** *"Incomplete sessions, meaning sessions that are not terminated by a request to Place Details (New) or Address Validation, are billed only for the Autocomplete (New) requests using the SKU: Autocomplete Requests"* — i.e., $2.83/1k after 10k free.

**Phase 5 billing implication:**
- Happy path (user picks a result, finishes a session): 1× Place Details Pro = **$17/1k** after 5k free.
- Abandoned autocomplete (user types but never picks): billed per autocomplete request, **$2.83/1k** after 10k free.
- Debounce and a minimum-character-length guard in Phase 5 are therefore cost-relevant, not just perf-relevant.

> **Caveat on stale pricing-page text:** The docs still reference a "$200 monthly credit valid until Feb 28 2025" in some sections. That is stale as of 2026-04-08; treat the per-SKU "Free Usage Cap" column as the authoritative free tier. If Phase 5 wants to surface a "free tier" number in the UI, it should re-check the pricing page at implementation time.

## What changed vs. the plan's initial assumptions

- The plan text assumed the Base UI install path might be unclear. Confirmed: it *is* at `@base-ui/react/combobox` (subpath, not package root). Plan language was correct.
- The plan's sample `matcher` (`"/((?!_next/static|_next/image|favicon.ico|.*\\.(?:png|jpg|svg)$).*)"`) was trimmed; the upstream Next.js Supabase example uses a slightly broader negative lookahead covering `svg|png|jpg|jpeg|gif|webp`. Phase 1 should use the broader form to avoid a latent gap for `.jpeg`/`.gif`/`.webp` passing through proxy.
- The plan assumed the Supabase docs page gives verbatim Next.js snippets. As of 2026-04-08, it does not — the Next.js snippets are only visible in the Next.js canary `with-supabase` example. The pre-adaptation canonical snippets recorded above come from that example, not from supabase.com.
- Google's doc pages do not publish a numeric session-token expiration. Phase 5 should rotate session tokens on each new "typing session" (e.g., on input clear, on successful selection, on mount) rather than timing them out.
